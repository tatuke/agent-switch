# Agent Soul Transfer Protocol (ASTP)
[English](README.md) | [简体中文](README.zh.md)

可移植的 AI 代理人格和身份、工作原则和安全守则转移系统。

## 背景信息
因为我在使用一些agent时，遇到更新agent 后整个工具无法启动和响应的情况，特别是一些更新很频繁。每次都包含了很多"改进"的项目。为了工作的可持续性，我不得不思考备份、恢复转移的可能性。同时兼顾硬件损坏后不得不转移的可能性。不仅仅是工作成果，还有我在长久和agent 工作过程中积累下来的"默契"。好吧，总的来说，是一些节省prompt 的定义、安全边界原则、代码规范、思考路径等。
所以我一开始打算开发一整套手动配置的架构来进行人工指导这个备份和转移的过程[INTRODUCE.md](./INTRODUCE.md)。
后来我觉得，为什么不让agent 之间自己约定一套通用的转移规范？当我使用三个agent 工具之间进行了多轮的讨论(opencode、claude code 、openclaw),并经过我个人的修改后，形成了这一套转移"规范"。这套规范是纯粹文字的，意味着你作为人类可以自己阅读做出定制化的修改。所以不用担心看不懂代码，也能掌握转移过程中的"大多数"进程。

## 给 AI Agent 看的介绍

该项目目前使用纯粹md文档作为转移过程中的说明和规范。 当需要打包你自己时，请阅读[packys_en.md](./packys_en.md)
当需要接收转移文件并更新自身时，请阅读[adaptys.md](./adaptys_en.md).
有必要在采取任何动作前，先向你的用户解释、总结项目文件中的功能和接下来的动作。如果得到允许执行，严格按照文件中的说明执行，未得到许可时，不要擅自做出任何删除自身规则文件或者工作文件的动作。

## 技术说明和一些后续计划
虽然我做出了暂时不更新代码架构部分的决定，但我仍然觉得有一部分需要代码来搭建部分架构，以使整个迁移过程更加通用和有序。本项目还存在以下问题。
1.现如今考虑到memory和skills 的总结和转移是一项稍显复杂和有巨大耗费的工程。我个人更倾向于使用内网skill [网络](https://github.com/tatuke/agent-skills-connect)和外接memory 组件，比如一些知识库和记忆库组件。
2.项目只经过了三个agent 工具之间的验证和测试，没在更多、更新的agent 组件上进行测试验证。
3.当agent工具更新(仓库源头)，又该如何适配和灵活调度。

这些以及后续问题会是未来代码解决的方向。
